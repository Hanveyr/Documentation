Comprehensive Guide to Linux File Permissions and User Management


-Introduction-
This guide provides a thorough explanation of Linux file permissions, including chown and chmod commands,
the permission bits and how they sum up for read, write, and execute permissions,
special permissions (SUID, SGID, Sticky), umask, and user management files such as /etc/passwd, /etc/shadow, /etc/skel,
and /etc/login.defs. This document is intended to help engineers understand and manage access permissions effectively.


Controlling Access to files with Permissions.


-Basic Concepts and Terminology-
    File Permissions: The access rights assigned to files and directories.
    Permission Bits: A set of bits that define the read, write, and execute permissions for the owner, group, and others.
    Special Permissions: Additional permissions like SUID, SGID, and Sticky bit that provide special access rights.
    umask: A default user mask that controls the permissions set for newly created files and directories.

---Understanding Permission Bits---


Summarized Table of Permission Bits:

| Octal | Binary | Permissions | Description                 |
|-------|--------|-------------|-----------------------------|
| 0     | 000    | ---         | No permissions              |
| 1     | 001    | --x         | Execute only                |
| 2     | 010    | -w-         | Write only                  |
| 3     | 011    | -wx         | Write and execute           |
| 4     | 100    | r--         | Read only                   |
| 5     | 101    | r-x         | Read and execute            |
| 6     | 110    | rw-         | Read and write              |
| 7     | 111    | rwx         | Read, write, and execute    |


-Read, Write, Execute Permissions (rwx)-

Each file and directory has three types of permissions for three different categories of users:
    Owner: The user who owns the file.
    Group: The group that owns the file.
    Others: All other users.

The permissions are represented by three characters each for the owner, group, and others:
    r: Read (4)
    w: Write (2)
    x: Execute (1)

The permissions can be represented numerically by summing the values of read, write, and execute:
    0: No permission (---)
    1: Execute (--x)
    2: Write (-w-)
    3: Write and Execute (-wx)
    4: Read (r--)
    5: Read and Execute (r-x)
    6: Read and Write (rw-)
    7: Read, Write, and Execute (rwx)


For example, 755 means:
    Owner: Read, Write, Execute (7)
    Group: Read, Execute (5)
    Others: Read, Execute (5)


Binary representation of permission bits:
    Owner (first three bits)
    Group (next three bits)
    Others (last three bits)

Permissions in binary:
    r: Read (100 in binary or 4)
    w: Write (010 in binary or 2)
    x: Execute (001 in binary or 1)


-Binary Representation of Permission Bits-
Permissions are also represented in binary format, where each bit corresponds to a specific permission:
    Owner: The first three bits
    Group: The next three bits
    Others: The last three bits

Each permission (read, write, execute) is represented as follows:
    r: Read (100 in binary or 4)
    w: Write (010 in binary or 2)
    x: Execute (001 in binary or 1)

For example, rwxr-xr-- translates to:
    Owner: rwx (111 in binary or 7)
    Group: r-x (101 in binary or 5)
    Others: r-- (100 in binary or 4)

Combined, this is represented as 755 in octal (binary: 111 101 100).






















Special bits (4 bits in total):

    Set User ID (SUID) - 4 (represented as s in owner's execute position)
    Set Group ID (SGID) - 2 (represented as s in group's execute position)
    Sticky Bit - 1 (represented as t in others' execute position)

----Understanding umask----


Umask (short for User file-creation mode Mask) is used by UNIX-based systems to set default permissions for newly created files and directories.

umask - Default, Opposite, Take Away, /etc/profile




-umask-
The umask command sets the default permissions for new files and directories. It subtracts the umask value from the default permissions (666 for files, 777 for directories).


Example: umask 022
Default permissions for files: 666
Resulting permissions: 666 - 022 = 644 (rw-r--r--)
Default permissions for directories: 777
Resulting permissions: 777 - 022 = 755 (rwxr-xr-x)


-Advanced umask Usage-

-umask: Using umask in shell scripts-
Set umask at the beginning of a shell script
umask 027
Create a file with specific permissions in a script
touch scriptfile

Special Permissions

-SUID, SGID, Sticky Bit-
    SUID (Set User ID): When set on an executable file, it allows the file to be executed with the privileges of the file owner. Represented as s in the owner's execute position.
    SGID (Set Group ID): When set on an executable file, it allows the file to be executed with the privileges of the file's group. When set on a directory, new files created within the directory inherit the group of the directory. Represented as s in the group's execute position.
    Sticky Bit: When set on a directory, it restricts file deletion within the directory to the file's owner. Represented as t in the others' execute position.

Special permissions can be set using the chmod command with an additional digit:
    4xxx: Set SUID
    2xxx: Set SGID
    1xxx: Set Sticky Bit

For example, 4755 means:
    SUID: Set User ID (4)
    Owner: Read, Write, Execute (7)
    Group: Read, Execute (5)
    Others: Read, Execute (5)

Special Permissions in Linux: Detailed Explanation and Common Use Cases

In Linux, special permissions provide additional access control features that go beyond the standard read, write, and execute permissions. These special permissions include the Sticky Bit, Set Group ID (SGID), and Set User ID (SUID).
Sticky Bit (Octal 1 / o+t)

The Sticky Bit, represented by 1 in the octal notation and o+t in symbolic form, is used primarily on directories. When the sticky bit is set on a directory, it restricts file deletion or renaming within that directory to the file owner, the directory owner, or the root user. This is particularly useful in shared directories where multiple users have write access, such as /tmp. For example, setting the sticky bit on /tmp ensures that users can only delete their own files, preventing others from deleting or renaming their files.

Command Example:
bash

chmod 1777 /tmp
chmod o+t /tmp

Common Uses:

    /tmp: Temporary files directory.
    /var/tmp: Temporary files persist across reboots.
    /home: User home directories.

Set Group ID (SGID) (Octal 2 / g+s)

The Set Group ID (SGID), represented by 2 in octal notation and g+s in symbolic form, has different effects on files and directories. When applied to an executable file, it allows the file to be executed with the group privileges of the file owner rather than the user's group. When set on a directory, SGID ensures that new files created within the directory inherit the directory's group ownership instead of the user's primary group. This is particularly useful for collaborative directories where group ownership is important.

Command Examples:

    On Files:
    bash

chmod 2755 /usr/bin/write
chmod g+s /usr/bin/write

On Directories:
bash

    chmod 2770 /home/shared
    chmod g+s /home/shared

Common Uses:

    /usr/bin/write: Write to other users' terminals.
    /var/www/html: Web files inherit www-data group.
    /home/shared: Team collaboration directories.

Set User ID (SUID) (Octal 4 / u+s)

The Set User ID (SUID), represented by 4 in octal notation and u+s in symbolic form, is used on executable files. When the SUID bit is set, the file runs with the privileges of the file owner, not the user who executed it. This is crucial for programs that need elevated privileges to perform certain operations, such as changing passwords or accessing raw sockets.

Command Example:
bash

chmod 4755 /usr/bin/passwd
chmod u+s /usr/bin/passwd

Common Uses:

    /usr/bin/passwd: Run as root to update /etc/shadow.
    /usr/bin/sudo: Run as root for privilege escalation.
    /usr/bin/ping: Access raw sockets as root.

Combined Special Permissions

Special permissions can be combined with regular permissions and other special permissions to create specific access control setups.

Examples:

    Full Collaboration Directory (SGID + wide open):
    bash

chmod 2777 /shared

Results: rwxrwsrwx

    Files inherit group.
    Everyone can read/write.

Secure Collaboration (SGID + restricted):
bash

chmod 2770 /team

Results: rwxrws---

    Files inherit group.
    Only group members access.

Protected Public Directory (Sticky + public):
bash

chmod 1755 /public

Results: rwxr-xr-t

    Everyone can read.
    Only owners can delete.

550 (r-xr-x---) # Group traverse/read only
511 (r-x--x--x) # Everyone traverse, owner read
510 (r-x--x---) # Group traverse, owner read
500 (r-x------) # Owner read/traverse only
444 (r--r--r--) # Everyone read only
440 (r--r-----) # Group read only
400 (r--------) # Owner read only

ERROR 1: SUID on shell scripts does not work - dangerous misconception
----------------------------------------------------------------

The original document states:
    "Setting SUID and SGID on a script for privileged access:"
    chmod 4755 /usr/local/bin/myscript.sh

This is WRONG and potentially dangerous to believe.

The Linux kernel deliberately ignores SUID and SGID on
interpreted scripts — any file that starts with a shebang
line (#! /bin/bash, #! /usr/bin/python3, etc.).

This was disabled decades ago as a security measure. A SUID
shell script creates a race condition (TOCTOU attack) between
when the kernel checks the script file and when the interpreter
reads it. An attacker can swap the file in that gap.

What actually happens:
    chmod 4755 /usr/local/bin/myscript.sh   # sets the bit
    ls -l /usr/local/bin/myscript.sh        # shows rwsr-xr-x
    # Looks like SUID is set. It is NOT honored at execution.
    # The script runs as the caller, not as the file owner.

The SUID bit IS honored on compiled binary executables:
    /usr/bin/passwd     # SUID binary, runs as root
    /usr/bin/sudo       # SUID binary, runs as root
    /usr/bin/ping       # SUID binary (older systems)
    These work because they are ELF binaries, not scripts.

If you need a script to run with elevated privileges:
    Option 1: Write a small C wrapper binary with SUID set.
    Option 2: Use sudo with a specific sudoers rule.
    Option 3: Use a systemd service running as a specific user.

How to verify a binary is ELF vs script:
    file /usr/bin/passwd    # ELF 64-bit LSB pie executable
    file /usr/bin/myscript  # Bourne-Again shell script
    # Only ELF binaries respect SUID


----------------------------------------------------------------
ERROR 2: umask is bitwise AND NOT, not subtraction
----------------------------------------------------------------

The original document describes umask as:
    666 - 022 = 644
    777 - 022 = 755

This is a convenient shortcut that works for common values
but is the WRONG mental model and breaks for other values.

The actual operation:
    new_permissions = base_permissions AND (NOT umask)

Worked example with umask 022:
    base for files:   110 110 110  (666 in octal)
    umask 022:        000 010 010
    NOT umask:        111 101 101
    AND result:       110 100 100  = 644 in octal

The subtraction shortcut works here (666-022=644) because
each octal digit never borrows from the next in normal cases.

Where subtraction FAILS:
    umask 037
    Subtraction: 666 - 037 = 629  <- invalid, digit 9 doesn't exist in octal
    AND model:
        base:    110 110 110
        ~037:    111 100 000
        result:  110 100 000  = 640  <- correct answer

    umask 026
    Subtraction: 777 - 026 = 751  <- plausible but is it right?
    AND model:
        base:    111 111 111
        ~026:    111 101 001
        result:  111 101 001  = 751  <- happens to match here

    umask 033
    For files:
    Subtraction: 666 - 033 = 633  <- invalid digit
    AND model:
        110 110 110
        ~033 = 111 100 100
        result: 110 100 100 = 644  <- correct

Always use the AND model to verify results. The subtraction
shortcut is a mnemonic, not the actual operation.

Practical verification:
    umask 037          # set a test umask
    touch testfile
    ls -l testfile     # check actual result
    rm testfile
    umask 022          # restore

SECTION 3 - THE s vs S AND t vs T DISTINCTION
================================================================

This is completely absent from the original and is one of the
most common points of confusion when reading ls -l output.

When a special bit (SUID, SGID, sticky) is set, it appears
in the execute position. But what if the execute bit is NOT set?

    lowercase s or t = special bit IS set, execute IS also set
    UPPERCASE S or T = special bit IS set, execute is NOT set

SUID examples:
    -rwsr-xr-x   SUID set, owner execute set    <- normal, works
    -rwSr-xr-x   SUID set, owner execute NOT set <- unusual, SUID is meaningless
                                                    (cannot execute = SUID irrelevant)

SGID examples:
    -rwxr-sr-x   SGID set, group execute set    <- normal
    -rwxr-Sr-x   SGID set, group execute NOT set <- unusual, pointless on a file

Sticky bit examples:
    drwxrwxrwt   sticky set, others execute set <- normal (/tmp)
    drwxrwxrwT   sticky set, others execute NOT set <- sticky works on dirs
                                                        regardless of execute bit
                                                        but unusual to see

How you get uppercase S:
    chmod u+s file          # if file has no execute bit, S appears
    chmod 4644 file         # 6=rw, 4=r, 4=r + SUID -> -rwSr--r--

    # Set SUID correctly (with execute):
    chmod 4755 file         # -rwsr-xr-x (correct)
    chmod 4644 file         # -rwSr--r-- (SUID set but useless)

Checking for this in practice:
    ls -l /usr/bin/passwd   # should show -rwsr-xr-x (lowercase s)
    ls -l /tmp              # should show drwxrwxrwt (lowercase t)

If you see uppercase S in a SUID binary it means the execute
bit is missing and the program cannot be run at all, let alone
with elevated privileges.

SECTION 4 - PRIMARY vs SUPPLEMENTARY GROUPS
================================================================

The original document covers group commands but never explains
the fundamental distinction that determines how permissions work.

Every user has exactly ONE primary group and zero or more
supplementary groups.

Primary group:
    - Stored as the GID field in /etc/passwd
    - Used as the group owner when you CREATE a new file
    - The group that appears in ls -l on files you create

Supplementary groups:
    - Listed in /etc/group for that user
    - Used for ACCESS CHECKS on existing files
    - You can be in many supplementary groups simultaneously

Example:
    alice's /etc/passwd entry: alice:x:1001:1001::/home/alice:/bin/bash
    alice's primary GID = 1001 (group "alice")

    /etc/group entries:
    alice:x:1001:
    developers:x:1005:alice,bob
    docker:x:999:alice

    alice creates a file:  -rw-r--r-- alice alice newfile.txt
    Group owner is "alice" (her primary group), not "developers"

    But alice CAN read files owned by group "developers",
    "docker", etc. because she is a supplementary member.

Checking your own groups:
    id                  # shows uid, gid, and all groups
    id alice            # shows groups for another user
    groups              # lists your group names
    groups alice        # lists groups for another user

Output of id:
    uid=1001(alice) gid=1001(alice) groups=1001(alice),999(docker),1005(developers)
    ^                ^               ^
    user ID          primary group   all groups (primary + supplementary)

When group membership changes take effect:
    If you add a user to a group with usermod -aG or gpasswd -a,
    the change does NOT take effect in existing sessions.
    The user must log out and log back in.

    To avoid logging out, use:
    newgrp developers       # starts a new shell with developers as active group
                            # your primary group becomes developers for this session
                            # exit to return to normal

    Or use:
    su - $USER              # re-login as yourself

Why this matters for permissions:
    A file owned by group "developers" with permissions 770:
    - alice can read/write it because she is in "developers" supplementary group
    - A new file alice creates will be owned by group "alice" (primary), NOT "developers"
    - To make alice create files owned by "developers", either:
      a. Use newgrp developers first
      b. Set SGID on the directory (new files inherit directory's group)
      c. Change alice's primary group: usermod -g developers alice

SECTION 7 - find WITH PERMISSION FILTERS (SECURITY AUDITING)

Finding SUID files (highest security concern):
    find / -perm -4000 -type f 2>/dev/null
    # -4000 means "SUID bit is set" (any other bits can be anything)
    # -type f means regular files only
    # 2>/dev/null suppresses permission denied messages

    # On a clean system you expect to see:
    # /usr/bin/passwd, /usr/bin/sudo, /usr/bin/su, /usr/bin/mount
    # Any unexpected SUID binary is a potential security issue

Finding SGID files:
    find / -perm -2000 -type f 2>/dev/null

Finding both SUID and SGID:
    find / -perm /6000 -type f 2>/dev/null
    # /6000 means "SUID OR SGID bit set" (/ means OR any bit)

Finding world-writable files (security risk):
    find / -perm -0002 -type f 2>/dev/null
    # -0002 means "others write bit is set"
    # World-writable files can be modified by anyone

Finding world-writable directories:
    find / -perm -0002 -type d 2>/dev/null

Finding files with no owner (orphaned):
    find / -nouser 2>/dev/null
    find / -nogroup 2>/dev/null
    # These appear after a user is deleted without cleaning up files

Finding files with exact permissions:
    find /etc -perm 644 -type f     # exactly 644, nothing else
    find /etc -perm 777 -type f     # exactly 777 (dangerous if found)

Finding files with AT LEAST these permissions:
    find /etc -perm -644 -type f    # 644 or more permissive
    find / -perm -u+s               # any file with SUID (symbolic form)

Finding files owned by a specific user:
    find /home -user alice -type f
    find / -uid 1001 -type f        # by UID number

Finding files owned by a specific group:
    find /var -group www-data -type f
    find / -gid 33 -type f          # by GID number

Full security audit one-liner:
    find / -type f \( -perm -4000 -o -perm -2000 \) -exec ls -la {} \; 2>/dev/null
    # Find all SUID and SGID files and show their full permissions


-perm flag syntax reference:
    -perm 755       exact match: permissions must be EXACTLY 755
    -perm -755      ALL listed bits must be set (755 is minimum)
    -perm /755      ANY listed bit must be set (755 or looser)
    -perm +755      same as / in newer find versions
    -perm -u+s      symbolic: user (owner) SUID bit must be set
    -perm /a+w      symbolic: any of owner/group/others write set

SECTION 9 - ACLs: FULL COVERAGE
================================================================

The original lists 5 ACL commands with no context. This covers
what ACLs are, when to use them, and how they actually work.

----------------------------------------------------------------
9a. Why ACLs exist
----------------------------------------------------------------

Standard Unix permissions only allow THREE permission sets:
owner, group, others. You cannot give a specific fourth user
access without adding them to the group (which gives all group
members access).

Example problem: /project is owned by alice:team with 770.
You want bob (not in "team") to have read access but no others.

Without ACLs: no way to do this cleanly.
With ACLs: setfacl -m u:bob:r-- /project

ACLs let you add fine-grained rules on top of standard permissions.
They are supported on ext4, xfs, btrfs and most modern filesystems.

Check if ACLs are enabled on a filesystem:
    tune2fs -l /dev/sda1 | grep "Default mount"   # should show "acl"
    mount | grep acl                               # check mount options

On most modern systems ACLs are enabled by default.


----------------------------------------------------------------
9b. Reading getfacl output
----------------------------------------------------------------

getfacl file.txt

Output:
    # file: file.txt
    # owner: alice
    # group: staff
    user::rw-          <- owner permissions (alice)
    user:bob:r--       <- ACL entry: bob gets read only
    group::r--         <- owning group permissions (staff)
    group:developers:rw- <- ACL entry: developers group gets rw
    mask::rw-          <- THE MASK (see below)
    other::r--         <- others permissions

The three types of ACL entries:
    user::rwx          owner (no name = file owner)
    user:name:rwx      a specific named user
    group::rwx         owning group (no name = file's group)
    group:name:rwx     a specific named group
    mask::rwx          the ACL mask (limits effective permissions)
    other::rwx         everyone else (same as chmod o=)


----------------------------------------------------------------
9c. The ACL mask - the most misunderstood part
----------------------------------------------------------------

The mask limits the effective permissions of ALL named users
and ALL groups (except the file owner and others).

    getfacl file.txt shows:
    user:bob:rwx      <- bob's entry says rwx
    mask::r--         <- mask is r-- (read only)
    # effective: r--  <- bob's effective permission is r--

    The mask is the CEILING. bob cannot exceed what the mask allows.
    Even though bob's entry says rwx, the mask limits him to r--.

When does the mask change?
    chmod g+w file    <- when you use chmod, it changes the MASK
                         not the group ACL entry

    This is why chmod can interfere with ACLs. When you run:
    chmod 644 file
    It sets mask to r-- (the group's traditional permission)
    This may reduce effective permissions for all ACL users.

    To change group traditional permissions WITHOUT touching the mask:
    setfacl -m group::r-- file     # change owning group entry
                                   # mask stays unchanged

View effective permissions (with mask applied):
    getfacl file.txt | grep "effective"
    # Lines with "# effective:" show what the mask actually allows


----------------------------------------------------------------
9d. Setting ACL entries
----------------------------------------------------------------

setfacl -m u:bob:rw file.txt       # give bob rw access
setfacl -m g:developers:rx file.txt # give developers group rx
setfacl -m o::--- file.txt         # remove others access
setfacl -m u:bob:- file.txt        # give bob no permissions (but entry exists)
setfacl -m mask::rwx file.txt      # set the mask to rwx (open ceiling)

Multiple entries at once:
    setfacl -m u:bob:rw,g:dev:rx,mask::rwx file.txt

Removing specific entries:
    setfacl -x u:bob file.txt          # remove bob's entry
    setfacl -x g:developers file.txt   # remove developers entry

Removing ALL ACLs:
    setfacl -b file.txt               # removes all ACL entries
                                       # reverts to standard permissions

Recursive:
    setfacl -R -m u:bob:rX /project   # X = execute only on directories


----------------------------------------------------------------
9e. Default ACLs on directories
----------------------------------------------------------------

Default ACLs are applied to new files/directories created inside
a directory. This is one of the most useful ACL features.

    setfacl -d -m u:bob:rw /project    # default ACL for new files
    setfacl -d -m g:dev:rwx /project   # default ACL for new dirs

    # -d flag means "set as default"

Now every file created inside /project automatically gets:
    user:bob:rw-   as an ACL entry

Viewing default ACLs:
    getfacl /project

    Output includes:
    default:user::rwx          <- default owner perms
    default:user:bob:rw-       <- bob gets rw on new files
    default:group::r-x
    default:mask::rwx
    default:other::r-x

Copy ACLs from one file to another:
    getfacl source.txt | setfacl --set-file=- target.txt

Backup and restore ACLs:
    getfacl -R /project > project_acls.txt    # backup
    setfacl --restore=project_acls.txt        # restore

SECTION 10 - FILE CAPABILITIES
================================================================

The original lists three capability commands. This explains
what capabilities actually are and why they matter.

----------------------------------------------------------------
10a. What capabilities are
----------------------------------------------------------------

Traditionally, privilege in Linux was binary: root (uid 0) has
everything, everyone else has nothing.

Capabilities break root's privileges into distinct units.
Instead of running a program as root just because it needs
one root privilege, you grant it only the specific capability
it needs.

Example: A web server needs to bind to port 80 (port < 1024
requires root). Instead of running nginx as root (dangerous),
grant it just the capability to bind low ports:

    setcap cap_net_bind_service=+ep /usr/sbin/nginx

Now nginx can bind port 80 while running as www-data (not root).

This is the principle of least privilege applied to programs.


----------------------------------------------------------------
10b. Capability sets
----------------------------------------------------------------

Each process has three capability sets:
    Permitted    capabilities the process is allowed to have
    Effective    capabilities currently active (kernel checks these)
    Inheritable  capabilities passed to child processes

When setting caps with setcap, the = notation sets which sets:
    +ep   add to Effective and Permitted
    +eip  add to Effective, Inheritable, and Permitted
    -ep   remove from Effective and Permitted

Common capabilities:
    cap_net_bind_service    bind to ports below 1024
    cap_net_raw             use raw sockets (ping)
    cap_sys_admin           many admin operations (broad, avoid)
    cap_dac_override        bypass file permission checks
    cap_chown               change file ownership
    cap_setuid              change UID (become other users)
    cap_kill                send signals to any process
    cap_net_admin           configure network interfaces
    cap_sys_ptrace          trace processes (needed by debuggers)
    cap_ipc_lock            lock memory (databases use this)


----------------------------------------------------------------
10c. Working with capabilities
----------------------------------------------------------------

View capabilities on a file:
    getcap /usr/sbin/nginx         # shows: /usr/sbin/nginx cap_net_bind_service=ep
    getcap /usr/bin/ping           # may show cap_net_raw=ep
    getcap /usr/bin/python3        # usually empty

Find all files with capabilities:
    getcap -r / 2>/dev/null        # recursive search from root
    # This is your capabilities audit command

Set a capability:
    setcap cap_net_bind_service=+ep /path/to/binary

Remove a specific capability:
    setcap cap_net_bind_service=-ep /path/to/binary

Remove ALL capabilities from a file:
    setcap -r /path/to/binary

View capabilities of running processes:
    cat /proc/PID/status | grep Cap
    # Shows CapInh, CapPrm, CapEff, CapBnd, CapAmb in hex
    # Decode with: capsh --decode=0000003fffffffff

SECTION 11 - EXTENDED FILE ATTRIBUTES (chattr/lsattr) IN DEPTH
================================================================

The original lists chattr flags. This explains what they do
and when to use them.

chattr flags that matter:

+i  immutable
    File cannot be modified, deleted, renamed, or linked.
    Even root cannot modify it without removing the flag first.
    Use: protect critical config files, /etc/passwd backups.

    chattr +i /etc/resolv.conf    # ISPs sometimes overwrite this
    chattr +i /etc/hosts          # prevent accidental overwrite
    lsattr /etc/resolv.conf       # shows: ----i---------e--- /etc/resolv.conf
    chattr -i /etc/resolv.conf    # remove immutable to edit

+a  append-only
    File can only be opened for appending, not overwriting.
    Perfect for log files -- even root cannot alter history.

    chattr +a /var/log/secure     # can write new lines, not modify old
    chattr +a /var/log/auth.log

+A  no atime updates
    Accessing the file does not update the access timestamp.
    Performance optimization -- prevents disk writes on every read.

+s  secure deletion
    When deleted, the blocks are zeroed. Prevents recovery.
    Note: on SSDs and journaling filesystems this is not
    guaranteed to work as intended. Use shred or cryptsetup
    for sensitive data instead.

+u  undeletable
    When deleted, kernel saves the contents for later recovery.
    Implementation depends on filesystem.

+e  extents
    File uses extents for block mapping (set automatically on ext4).

+C  no copy-on-write
    On btrfs and similar filesystems, disables COW for this file.
    Used for virtual machine disk images to reduce fragmentation.

+c  compressed
    File is automatically compressed by the kernel.
    Filesystem must support compression (btrfs, etc.).

Viewing attributes:
    lsattr file.txt            # show attributes for one file
    lsattr -R /etc             # recursive
    lsattr -d /etc             # show directory attributes (not contents)

Output format:
    ----i---------e--- /etc/resolv.conf
    The dashes are positions for each attribute flag.
    i = immutable, e = extents, all others empty here.

Important security note:
    chattr +i is not visible to standard tools (ls, stat).
    A file can look deletable but chattr +i prevents it.
    Always check lsattr when you cannot delete/modify a file
    even as root.

    # "Why can't I delete this file even as root?"
    lsattr /path/to/file       # check for +i flag


================================================================
SECTION 12 - PROCESS PERMISSIONS AND EFFECTIVE UID
================================================================

Understanding why SUID works requires understanding how the
kernel checks permissions during process execution.

----------------------------------------------------------------
12a. Three UIDs every process has
----------------------------------------------------------------

Every running process has three user IDs:

Real UID (RUID):
    Who actually started the process. Your login UID.
    Stays constant throughout the process's life normally.

Effective UID (EUID):
    What the kernel uses for permission checks.
    Normally equals the real UID.
    Changed by SUID: when you run a SUID binary, EUID becomes
    the file owner's UID.

Saved Set-UID (SUID in process context):
    Saved copy of the original EUID before a privilege change.
    Allows a process to drop and regain privilege.

Example: running passwd (SUID root)
    alice runs /usr/bin/passwd
    RUID = 1001 (alice's UID)  <- who really started it
    EUID = 0    (root)         <- file owner's UID (SUID kicks in)
    passwd can now write to /etc/shadow because kernel sees EUID=0

Check a process's UIDs:
    cat /proc/PID/status | grep -E "^(R|E|S)uid"
    ps -o pid,ruid,euid,suid,comm

----------------------------------------------------------------
12b. How the kernel checks permissions
----------------------------------------------------------------

When a process tries to access a file, the kernel:

    1. Is EUID == 0 (root)?  YES -> allow everything
    2. Is EUID == file owner UID?  YES -> apply owner rwx bits
    3. Is any of process's GIDs == file's GID?  YES -> apply group rwx bits
    4. Apply others rwx bits

The kernel checks these in order and stops at the first match.

Important implication:
    If you OWN a file but have permission 007 (---rwxrwx),
    the owner bits (---) apply to you and you cannot read
    your own file, even though others can.

    chmod 007 myfile.txt
    cat myfile.txt     # Permission denied - you own it but 007 denies owner

    This trips up people who think "I own it so I can do anything."
    The kernel applies ownership category, not logic.

----------------------------------------------------------------
12c. Checking effective vs real UID in scripts
----------------------------------------------------------------

In a script, check if running as root:
    if [ "$(id -u)" -ne 0 ]; then
        echo "Must be run as root"
        exit 1
    fi

    # id -u returns EUID (effective), so this catches SUID escalation too

Check if SUID elevated your script (it won't, but shows the concept):
    echo "Real UID: $(id -ru)"        # real UID
    echo "Effective UID: $(id -u)"    # effective UID

SECTION 13 - chmod --reference AND chgrp

Two commands that belong here and are absent.

chmod --reference:
    Copy permissions from one file to another.

    chmod --reference=template.sh newscript.sh
    # newscript.sh gets exactly the same permissions as template.sh

    stat template.sh    # shows 0755
    chmod --reference=template.sh newscript.sh

chgrp - change group owner only:

    chgrp is the dedicated command for the same thing.

    chown :developers file.txt      # same result
    chgrp developers file.txt       # same result

SECTION 14 - SECURITY AUDITING WITH PERMISSIONS
================================================================

Practical commands for finding permission problems on a system.

Find all SUID binaries (run periodically and compare):
    find / -perm -4000 -type f 2>/dev/null | sort > /tmp/suid_audit.txt
    diff /tmp/suid_baseline.txt /tmp/suid_audit.txt
    # Any new SUID binary since baseline is suspicious

Find world-writable files outside /tmp:
    find / -perm -0002 -type f ! -path "/proc/*" ! -path "/tmp/*" 2>/dev/null

Find files with no owner or group:
    find / -nouser -o -nogroup 2>/dev/null
    # These should not exist on a clean system
    # Appear when users are deleted without cleaning up files

    find / -user root -perm -0002 -type f 2>/dev/null
    # Root-owned but world-writable = serious problem

Check all home directories have correct ownership:
    ls -la /home/              # all should be owned by their user
    stat /home/alice           # verify permissions are 700 or 750

Find recently modified SUID files (potential compromise):
    find / -perm -4000 -newer /var/log/syslog 2>/dev/null

List all files alice owns:
    find / -user alice 2>/dev/null

Verify /etc/passwd and /etc/shadow permissions:
    ls -la /etc/passwd     # should be -rw-r--r-- root:root 644
    ls -la /etc/shadow     # should be -r-------- root:shadow or 640
    ls -la /etc/sudoers    # should be -r--r----- root:root 440

SECTION 16 - COMMON PERMISSION PATTERNS AND WHAT THEY MEAN
================================================================

Reference for understanding permissions you encounter in the wild.

Files:
    -rw-------  600  owner read/write only. SSH private keys, root files.
    -rw-r--r--  644  standard file. Everyone reads, only owner writes.
    -rw-rw-r--  664  group can write. Collaborative files.
    -rw-rw-rw-  666  everyone can write. Temporary shared files. Use carefully.
    -rwx------  700  owner executable only. Personal scripts.
    -rwxr-xr-x  755  standard executable. Everyone can run it.
    -rwxrwxrwx  777  everyone can do everything. Almost never correct for files.
    -r--------  400  read-only even for owner. Backup files, old logs.
    -r--r--r--  444  everyone read-only. Published static files.

Directories:
    drwx------  700  private. Owner only. Typical user home (/home/user).
    drwxr-x---  750  owner full, group read/traverse. Shared with group only.
    drwxr-xr-x  755  standard directory. Everyone can list and traverse.
    drwxrwsr-x  2775 SGID + group write. Shared project directories.
    drwxrwxrwt  1777 World-writable with sticky. /tmp pattern.

Special files you'll see in ls -l:
    -rwsr-xr-x  4755  SUID. Runs as file owner (root usually).
    -rwxr-sr-x  2755  SGID on file. Runs as file's group.
    drwxrwsrwt  3777  SGID dir + sticky. Collaborative + protected.

Real system examples:
    /etc/passwd         -rw-r--r-- (644) root:root
    /etc/shadow         ---------- (000) or -r-------- (400) root:shadow
    /etc/sudoers        -r--r----- (440) root:root
    /home/alice         drwx------ (700) alice:alice
    /tmp                drwxrwxrwt (1777) root:root
    /usr/bin/passwd     -rwsr-xr-x (4755) root:root
    /usr/bin/sudo       -rwsr-xr-x (4755) root:root
    ~/.ssh/             drwx------ (700) user:user   MUST be 700
    ~/.ssh/authorized_keys -rw------- (600) user:user MUST be 600
    ~/.ssh/id_rsa       -rw------- (600) user:user   MUST be 600
    ~/.ssh/id_rsa.pub   -rw-r--r-- (644) user:user

SSH permission requirements are enforced by sshd:
    If ~/.ssh/ or key files have wrong permissions, SSH refuses
    to use them for security reasons.
    sshd will log: "bad permissions: ignore key file"

SECTION 17 - QUICK PERMISSION DEBUGGING WORKFLOW
================================================================

When you get "Permission denied" and need to fix it fast.

Step 1: Identify what is failing
    ls -la /path/to/file           # check the file itself

Step 2: Check all parent directories
    namei -l /path/to/file         # check entire path
    # Look for any directory that blocks your access

Step 3: Confirm your identity
    id                             # confirm your UID and groups
    # Newly added to a group? You need to log out and back in.

Step 4: Check for special blockers
    lsattr /path/to/file           # check for +i (immutable)
    getfacl /path/to/file          # check for ACL overrides

Step 5: Check process's effective UID (if a service is failing)
    systemctl status servicename   # check what user service runs as
    cat /proc/PID/status | grep Uid  # check running process's UID

Common fixes:
    # Wrong file ownership:
    chown correctuser:correctgroup /path/to/file

    # Wrong directory permissions:
    chmod 755 /path/to/directory

    # User not in required group:
    usermod -aG groupname username
    # (user must re-login after)

    # File is immutable:
    chattr -i /path/to/file        # remove immutable
    # then make your change
    chattr +i /path/to/file        # re-apply if needed

    # ACL is blocking:
    getfacl /path/to/file          # read what the ACL says
    setfacl -m u:yourusername:rw /path/to/file  # add yourself
    # or
    setfacl -b /path/to/file       # remove all ACLs, revert to standard

SECTION 1 - CORRECTIONS TO ERRORS IN THE ORIGINAL DOCUMENT

----------------------------------------------------------------
ERROR 1: SUID on shell scripts does not work - dangerous misconception
----------------------------------------------------------------

The original document states:
    "Setting SUID and SGID on a script for privileged access:"
    chmod 4755 /usr/local/bin/myscript.sh

This is WRONG and potentially dangerous to believe.

The Linux kernel deliberately ignores SUID and SGID on
interpreted scripts — any file that starts with a shebang
line (#! /bin/bash, #! /usr/bin/python3, etc.).

This was disabled decades ago as a security measure. A SUID
shell script creates a race condition (TOCTOU attack) between
when the kernel checks the script file and when the interpreter
reads it. An attacker can swap the file in that gap.

What actually happens:
    chmod 4755 /usr/local/bin/myscript.sh   # sets the bit
    ls -l /usr/local/bin/myscript.sh        # shows rwsr-xr-x
    # Looks like SUID is set. It is NOT honored at execution.
    # The script runs as the caller, not as the file owner.

The SUID bit IS honored on compiled binary executables:
    /usr/bin/passwd     # SUID binary, runs as root
    /usr/bin/sudo       # SUID binary, runs as root
    /usr/bin/ping       # SUID binary (older systems)
    These work because they are ELF binaries, not scripts.

If you need a script to run with elevated privileges:
    Option 1: Write a small C wrapper binary with SUID set.
    Option 2: Use sudo with a specific sudoers rule.
    Option 3: Use a systemd service running as a specific user.

How to verify a binary is ELF vs script:
    file /usr/bin/passwd    # ELF 64-bit LSB pie executable
    file /usr/bin/myscript  # Bourne-Again shell script
    # Only ELF binaries respect SUID


----------------------------------------------------------------
ERROR 2: umask is bitwise AND NOT, not subtraction
----------------------------------------------------------------

The original document describes umask as:
    666 - 022 = 644
    777 - 022 = 755

This is a convenient shortcut that works for common values
but is the WRONG mental model and breaks for other values.

The actual operation:
    new_permissions = base_permissions AND (NOT umask)

Worked example with umask 022:
    base for files:   110 110 110  (666 in octal)
    umask 022:        000 010 010
    NOT umask:        111 101 101
    AND result:       110 100 100  = 644 in octal

The subtraction shortcut works here (666-022=644) because
each octal digit never borrows from the next in normal cases.

Where subtraction FAILS:
    umask 037
    Subtraction: 666 - 037 = 629  <- invalid, digit 9 doesn't exist in octal
    AND model:
        base:    110 110 110
        ~037:    111 100 000
        result:  110 100 000  = 640  <- correct answer

    umask 026
    Subtraction: 777 - 026 = 751  <- plausible but is it right?
    AND model:
        base:    111 111 111
        ~026:    111 101 001
        result:  111 101 001  = 751  <- happens to match here

    umask 033
    For files:
    Subtraction: 666 - 033 = 633  <- invalid digit
    AND model:
        110 110 110
        ~033 = 111 100 100
        result: 110 100 100 = 644  <- correct

Always use the AND model to verify results. The subtraction
shortcut is a mnemonic, not the actual operation.

Practical verification:
    umask 037          # set a test umask
    touch testfile
    ls -l testfile     # check actual result
    rm testfile
    umask 022          # restore

SECTION 2 - READING ls -l OUTPUT IN FULL DETAIL
================================================================

The original document covers permission bits but never explains

Example ls -l line:
    -rwxr-xr-x  2  alice  staff  4096  Jan 15 10:30  script.sh
    ^          ^ ^  ^     ^     ^     ^               ^
    |          | |  |     |     |     |               filename
    |          | |  |     |     |     last modified
    |          | |  |     |     file size in bytes
    |          | |  |     group owner
    |          | |  user owner
    |          | hard link count
    |          permission string (10 chars)
    file type

The 10-character permission string:
    Position 1:    file type
    Positions 2-4: owner permissions (rwx)
    Positions 5-7: group permissions (rwx)
    Positions 8-10: others permissions (rwx)

File type characters (position 1):
    -    regular file
    d    directory
    l    symbolic link
    b    block device (/dev/sda)
    c    character device (/dev/tty)
    p    named pipe (FIFO)
    s    socket file

Full examples with explanation:
    -rw-r--r--   regular file, owner rw, group r, others r
    drwxr-xr-x   directory, owner rwx, group rx, others rx
    lrwxrwxrwx   symlink (permissions on symlink itself, ignored)
    -rwsr-xr-x   regular file with SUID (s in owner execute position)
    -rwxr-sr-x   regular file with SGID (s in group execute position)
    drwxrwxrwt   directory with sticky bit (t in others execute position)
    -rwSr--r--   SUID set but execute NOT set (S uppercase = warning)
    -rwxr-Sr--   SGID set but group execute NOT set (S uppercase)
    drwxrwxrwT   sticky set but others execute NOT set (T uppercase)


================================================================
SECTION 3 - THE s vs S AND t vs T DISTINCTION
================================================================

This is completely absent from the original and is one of the
most common points of confusion when reading ls -l output.

When a special bit (SUID, SGID, sticky) is set, it appears
in the execute position. But what if the execute bit is NOT set?

    lowercase s or t = special bit IS set, execute IS also set
    UPPERCASE S or T = special bit IS set, execute is NOT set

SUID examples:
    -rwsr-xr-x   SUID set, owner execute set    <- normal, works
    -rwSr-xr-x   SUID set, owner execute NOT set <- unusual, SUID is meaningless
                                                    (cannot execute = SUID irrelevant)

SGID examples:
    -rwxr-sr-x   SGID set, group execute set    <- normal
    -rwxr-Sr-x   SGID set, group execute NOT set <- unusual, pointless on a file

Sticky bit examples:
    drwxrwxrwt   sticky set, others execute set <- normal (/tmp)
    drwxrwxrwT   sticky set, others execute NOT set <- sticky works on dirs
                                                        regardless of execute bit
                                                        but unusual to see

How you get uppercase S:
    chmod u+s file          # if file has no execute bit, S appears
    chmod 4644 file         # 6=rw, 4=r, 4=r + SUID -> -rwSr--r--

    # Set SUID correctly (with execute):
    chmod 4755 file         # -rwsr-xr-x (correct)
    chmod 4644 file         # -rwSr--r-- (SUID set but useless)

Checking for this in practice:
    ls -l /usr/bin/passwd   # should show -rwsr-xr-x (lowercase s)
    ls -l /tmp              # should show drwxrwxrwt (lowercase t)

If you see uppercase S in a SUID binary it means the execute
bit is missing and the program cannot be run at all, let alone
with elevated privileges.


================================================================
SECTION 4 - PRIMARY vs SUPPLEMENTARY GROUPS
================================================================

The original document covers group commands but never explains
the fundamental distinction that determines how permissions work.

Every user has exactly ONE primary group and zero or more
supplementary groups.

Primary group:
    - Stored as the GID field in /etc/passwd
    - Used as the group owner when you CREATE a new file
    - The group that appears in ls -l on files you create

Supplementary groups:
    - Listed in /etc/group for that user
    - Used for ACCESS CHECKS on existing files
    - You can be in many supplementary groups simultaneously

Example:
    alice's /etc/passwd entry: alice:x:1001:1001::/home/alice:/bin/bash
    alice's primary GID = 1001 (group "alice")

    /etc/group entries:
    alice:x:1001:
    developers:x:1005:alice,bob
    docker:x:999:alice

    alice creates a file:  -rw-r--r-- alice alice newfile.txt
    Group owner is "alice" (her primary group), not "developers"

    But alice CAN read files owned by group "developers",
    "docker", etc. because she is a supplementary member.

Checking your own groups:
    id                  # shows uid, gid, and all groups
    id alice            # shows groups for another user
    groups              # lists your group names
    groups alice        # lists groups for another user

Output of id:
    uid=1001(alice) gid=1001(alice) groups=1001(alice),999(docker),1005(developers)
    ^                ^               ^
    user ID          primary group   all groups (primary + supplementary)

When group membership changes take effect:
    If you add a user to a group with usermod -aG or gpasswd -a,
    the change does NOT take effect in existing sessions.
    The user must log out and log back in.

    To avoid logging out, use:
    newgrp developers       # starts a new shell with developers as active group
                            # your primary group becomes developers for this session
                            # exit to return to normal

    Or use:
    su - $USER              # re-login as yourself

Why this matters for permissions:
    A file owned by group "developers" with permissions 770:
    - alice can read/write it because she is in "developers" supplementary group
    - A new file alice creates will be owned by group "alice" (primary), NOT "developers"
    - To make alice create files owned by "developers", either:
      a. Use newgrp developers first
      b. Set SGID on the directory (new files inherit directory's group)
      c. Change alice's primary group: usermod -g developers alice

SECTION 5 - newgrp EXPLAINED IN CONTEXT
================================================================

Listed in the original without explanation.

newgrp groupname    # change your active primary group for this session

What it does:
    Starts a new shell with the specified group as your primary group.
    New files created in that shell will be owned by that group.
    You must already be a member of the group.

    alice$ id
    uid=1001(alice) gid=1001(alice) groups=1001(alice),1005(developers)

    alice$ newgrp developers
    alice$ id
    uid=1001(alice) gid=1005(developers) groups=1001(alice),1005(developers)

    alice$ touch newfile.txt
    alice$ ls -l newfile.txt
    -rw-r--r-- 1 alice developers 0 Jan 15 10:30 newfile.txt
    #                   ^^^^^^^^^^
    #                   now owned by developers group

    alice$ exit        # return to original shell, primary group back to alice

newgrp with no argument:
    newgrp             # return to your primary group (resets to default)

newgrp with a password:
    If a group has a password set (rare), newgrp will prompt for it.
    This allows non-members to temporarily join a group.
    Group passwords are stored in /etc/gshadow.
    gpasswd groupname   # set a group password

SECTION 6 - DIAGNOSING PERMISSION DENIED ERRORS
================================================================

The original teaches how to SET permissions but not how to
DIAGNOSE when they go wrong. This is what you need on the job.

The most common mistake: you fix permissions on a file but
the error persists because a PARENT DIRECTORY is blocking access.

----------------------------------------------------------------
6a. namei - trace permissions along a path
----------------------------------------------------------------

namei -l /path/to/file

Shows the permissions of every component along the path.
Use this when you get "Permission denied" and can't see why.

    namei -l /var/log/nginx/access.log

    Output:
    f: /var/log/nginx/access.log
    dr-xr-xr-x root  root  /
    drwxr-xr-x root  root  var
    drwxr-xr-x root  root  log
    drwx------ root  root  nginx        <- HERE: others have no access
    -rw-r--r-- root  adm   access.log

    Even though access.log is readable, /var/log/nginx/ is 700 (root only).
    A non-root user cannot enter the directory, so cannot read the file.

    Fix:
    chmod 755 /var/log/nginx       # allow others to enter and read
    # or
    chmod 750 /var/log/nginx       # allow group to enter, not others
    chown root:adm /var/log/nginx  # then add user to adm group

namei without -l (just shows path components):
    namei /etc/ssh/sshd_config     # shows each component without permissions


----------------------------------------------------------------
6b. stat - detailed file information
----------------------------------------------------------------

stat file.txt     # shows everything about a file

Output:
    File: file.txt
    Size: 1234      Blocks: 8     IO Block: 4096  regular file
    Device: 802h/2050d  Inode: 131073  Links: 1
    Access: (0644/-rw-r--r--)  Uid: ( 1001/ alice)  Gid: ( 1001/ alice)
    Access: 2024-01-15 10:30:00.000
    Modify: 2024-01-14 09:15:00.000
    Change: 2024-01-14 09:15:00.000
     Birth: -

    The Access line shows BOTH octal (0644) AND symbolic (-rw-r--r--)
    Links: 1 means one hard link (no other names for this inode)
    Inode number is shown

stat on a directory:
    stat /etc/ssh          # shows permissions on the directory itself

stat vs ls:
    ls -l file    # quick, shows permissions
    stat file     # detailed, shows inode, all timestamps, block count


----------------------------------------------------------------
6c. id - your identity and group memberships
----------------------------------------------------------------

id                    # show your uid, gid, all groups
id username           # show another user's uid, gid, groups
id -u                 # just your UID number
id -g                 # just your primary GID number
id -G                 # all your GIDs (space separated numbers)
id -un                # just your username
id -gn                # just your primary group name
id -Gn                # all your group names

When to use it:
    Permission denied unexpectedly? Run id to confirm what
    groups you are actually in. Newly added groups require
    re-login to appear.

    id | grep docker    # check if you are in the docker group
    id | grep sudo      # check if you are in the sudo group


----------------------------------------------------------------
6d. groups - list group memberships
----------------------------------------------------------------

groups              # list your groups by name
groups alice        # list alice's groups by name

Difference from id:
    id shows UIDs and GIDs (numbers and names, all details)
    groups shows only group names, simpler output

Common permission debugging workflow:
    1. stat /path/to/file          # check permissions on the file
    2. namei -l /path/to/file      # check all parent directories
    3. id                          # confirm your groups
    4. ls -ld /path/to/directory   # check specific directory

SECTION 7 - find WITH PERMISSION FILTERS (SECURITY AUDITING)
================================================================

The original covers find in its own file (Find.txt) but the
permission-specific flags belong here and are completely absent.

These are essential for security auditing and RHCSA/LFCS exams.

Finding SUID files (highest security concern):
    find / -perm -4000 -type f 2>/dev/null
    # -4000 means "SUID bit is set" (any other bits can be anything)
    # -type f means regular files only
    # 2>/dev/null suppresses permission denied messages

    # On a clean system you expect to see:
    # /usr/bin/passwd, /usr/bin/sudo, /usr/bin/su, /usr/bin/mount
    # Any unexpected SUID binary is a potential security issue

Finding SGID files:
    find / -perm -2000 -type f 2>/dev/null

Finding both SUID and SGID:
    find / -perm /6000 -type f 2>/dev/null
    # /6000 means "SUID OR SGID bit set" (/ means OR any bit)

Finding world-writable files (security risk):
    find / -perm -0002 -type f 2>/dev/null
    # -0002 means "others write bit is set"
    # World-writable files can be modified by anyone

Finding world-writable directories:
    find / -perm -0002 -type d 2>/dev/null

Finding files with no owner (orphaned):
    find / -nouser 2>/dev/null
    find / -nogroup 2>/dev/null
    # These appear after a user is deleted without cleaning up files

Finding files with exact permissions:
    find /etc -perm 644 -type f     # exactly 644, nothing else
    find /etc -perm 777 -type f     # exactly 777 (dangerous if found)

Finding files with AT LEAST these permissions:
    find /etc -perm -644 -type f    # 644 or more permissive
    find / -perm -u+s               # any file with SUID (symbolic form)

Finding files owned by a specific user:
    find /home -user alice -type f
    find / -uid 1001 -type f        # by UID number

Finding files owned by a specific group:
    find /var -group www-data -type f
    find / -gid 33 -type f          # by GID number

Full security audit one-liner:
    find / -type f \( -perm -4000 -o -perm -2000 \) -exec ls -la {} \; 2>/dev/null
    # Find all SUID and SGID files and show their full permissions


-perm flag syntax reference:
    -perm 755       exact match: permissions must be EXACTLY 755
    -perm -755      ALL listed bits must be set (755 is minimum)
    -perm /755      ANY listed bit must be set (755 or looser)
    -perm +755      same as / in newer find versions
    -perm -u+s      symbolic: user (owner) SUID bit must be set
    -perm /a+w      symbolic: any of owner/group/others write set

SECTION 8 - chage: PASSWORD AGING MANAGEMENT
================================================================

The original mentions /etc/shadow fields for password aging
but never covers chage, the tool for managing them.

chage -l username       # list password aging info for a user
chage username          # interactive mode to set all aging settings

Example output of chage -l alice:
    Last password change                    : Jan 15, 2024
    Password expires                        : never
    Password inactive                       : never
    Account expires                         : never
    Minimum number of days between change   : 0
    Maximum number of days between change   : 99999
    Number of days of warning before expiry : 7

Setting aging parameters:
    chage -M 90 alice        # password expires after 90 days
    chage -m 7 alice         # must wait 7 days before changing again
    chage -W 14 alice        # warn 14 days before expiry
    chage -I 30 alice        # lock account 30 days after password expires
    chage -E 2024-12-31 alice # account expires on this date
    chage -E -1 alice        # remove account expiry (never expires)
    chage -d 0 alice         # force password change on next login

    chage -d 0 alice         # sets last change to epoch (0 days since 1970)
                             # forces change at next login

    for user in $(cut -d: -f1 /etc/passwd); do
        echo "$user:"; chage -l $user | grep expires
    done

    chage -M 90    sets the "max" field in /etc/shadow

    chage -W 14    sets the "warn" field
    chage -I 30    sets the "inactive" field
    chage -E date  sets the "expire" field
    chage -d 0     sets the "last_change" field to 0

SECTION 9 - ACLs: FULL COVERAGE
================================================================

The original lists 5 ACL commands with no context. This covers
what ACLs are, when to use them, and how they actually work.

----------------------------------------------------------------
9a. Why ACLs exist
----------------------------------------------------------------

Standard Unix permissions only allow THREE permission sets:
owner, group, others. You cannot give a specific fourth user
access without adding them to the group (which gives all group
members access).

Example problem: /project is owned by alice:team with 770.
You want bob (not in "team") to have read access but no others.

Without ACLs: no way to do this cleanly.
With ACLs: setfacl -m u:bob:r-- /project

ACLs let you add fine-grained rules on top of standard permissions.
They are supported on ext4, xfs, btrfs and most modern filesystems.

Check if ACLs are enabled on a filesystem:
    tune2fs -l /dev/sda1 | grep "Default mount"   # should show "acl"
    mount | grep acl                               # check mount options

On most modern systems ACLs are enabled by default.


----------------------------------------------------------------
9b. Reading getfacl output
----------------------------------------------------------------

getfacl file.txt

Output:
    # file: file.txt
    # owner: alice
    # group: staff
    user::rw-          <- owner permissions (alice)
    user:bob:r--       <- ACL entry: bob gets read only
    group::r--         <- owning group permissions (staff)
    group:developers:rw- <- ACL entry: developers group gets rw
    mask::rw-          <- THE MASK (see below)
    other::r--         <- others permissions

The three types of ACL entries:
    user::rwx          owner (no name = file owner)
    user:name:rwx      a specific named user
    group::rwx         owning group (no name = file's group)
    group:name:rwx     a specific named group
    mask::rwx          the ACL mask (limits effective permissions)
    other::rwx         everyone else (same as chmod o=)


----------------------------------------------------------------
9c. The ACL mask - the most misunderstood part
----------------------------------------------------------------

The mask limits the effective permissions of ALL named users
and ALL groups (except the file owner and others).

    getfacl file.txt shows:
    user:bob:rwx      <- bob's entry says rwx
    mask::r--         <- mask is r-- (read only)
    # effective: r--  <- bob's effective permission is r--

    The mask is the CEILING. bob cannot exceed what the mask allows.
    Even though bob's entry says rwx, the mask limits him to r--.

When does the mask change?
    chmod g+w file    <- when you use chmod, it changes the MASK
                         not the group ACL entry

    This is why chmod can interfere with ACLs. When you run:
    chmod 644 file
    It sets mask to r-- (the group's traditional permission)
    This may reduce effective permissions for all ACL users.

    To change group traditional permissions WITHOUT touching the mask:
    setfacl -m group::r-- file     # change owning group entry
                                   # mask stays unchanged

View effective permissions (with mask applied):
    getfacl file.txt | grep "effective"
    # Lines with "# effective:" show what the mask actually allows


----------------------------------------------------------------
9d. Setting ACL entries
----------------------------------------------------------------

setfacl -m u:bob:rw file.txt       # give bob rw access
setfacl -m g:developers:rx file.txt # give developers group rx
setfacl -m o::--- file.txt         # remove others access
setfacl -m u:bob:- file.txt        # give bob no permissions (but entry exists)
setfacl -m mask::rwx file.txt      # set the mask to rwx (open ceiling)

Multiple entries at once:
    setfacl -m u:bob:rw,g:dev:rx,mask::rwx file.txt

Removing specific entries:
    setfacl -x u:bob file.txt          # remove bob's entry
    setfacl -x g:developers file.txt   # remove developers entry

Removing ALL ACLs:
    setfacl -b file.txt               # removes all ACL entries
                                       # reverts to standard permissions

Recursive:
    setfacl -R -m u:bob:rX /project   # X = execute only on directories


----------------------------------------------------------------
9e. Default ACLs on directories
----------------------------------------------------------------

Default ACLs are applied to new files/directories created inside
a directory. This is one of the most useful ACL features.

    setfacl -d -m u:bob:rw /project    # default ACL for new files
    setfacl -d -m g:dev:rwx /project   # default ACL for new dirs

    # -d flag means "set as default"

Now every file created inside /project automatically gets:
    user:bob:rw-   as an ACL entry

Viewing default ACLs:
    getfacl /project

    Output includes:
    default:user::rwx          <- default owner perms
    default:user:bob:rw-       <- bob gets rw on new files
    default:group::r-x
    default:mask::rwx
    default:other::r-x

Copy ACLs from one file to another:
    getfacl source.txt | setfacl --set-file=- target.txt

Backup and restore ACLs:
    getfacl -R /project > project_acls.txt    # backup
    setfacl --restore=project_acls.txt        # restore

SECTION 10 - FILE CAPABILITIES
================================================================

The original lists three capability commands. This explains
what capabilities actually are and why they matter.

----------------------------------------------------------------
10a. What capabilities are
----------------------------------------------------------------

Traditionally, privilege in Linux was binary: root (uid 0) has
everything, everyone else has nothing.

Capabilities break root's privileges into distinct units.
Instead of running a program as root just because it needs
one root privilege, you grant it only the specific capability
it needs.

Example: A web server needs to bind to port 80 (port < 1024
requires root). Instead of running nginx as root (dangerous),
grant it just the capability to bind low ports:

    setcap cap_net_bind_service=+ep /usr/sbin/nginx

Now nginx can bind port 80 while running as www-data (not root).

This is the principle of least privilege applied to programs.


----------------------------------------------------------------
10b. Capability sets
----------------------------------------------------------------

Each process has three capability sets:
    Permitted    capabilities the process is allowed to have
    Effective    capabilities currently active (kernel checks these)
    Inheritable  capabilities passed to child processes

When setting caps with setcap, the = notation sets which sets:
    +ep   add to Effective and Permitted
    +eip  add to Effective, Inheritable, and Permitted
    -ep   remove from Effective and Permitted

Common capabilities:
    cap_net_bind_service    bind to ports below 1024
    cap_net_raw             use raw sockets (ping)
    cap_sys_admin           many admin operations (broad, avoid)
    cap_dac_override        bypass file permission checks
    cap_chown               change file ownership
    cap_setuid              change UID (become other users)
    cap_kill                send signals to any process
    cap_net_admin           configure network interfaces
    cap_sys_ptrace          trace processes (needed by debuggers)
    cap_ipc_lock            lock memory (databases use this)


----------------------------------------------------------------
10c. Working with capabilities
----------------------------------------------------------------

View capabilities on a file:
    getcap /usr/sbin/nginx         # shows: /usr/sbin/nginx cap_net_bind_service=ep
    getcap /usr/bin/ping           # may show cap_net_raw=ep
    getcap /usr/bin/python3        # usually empty

Find all files with capabilities:
    getcap -r / 2>/dev/null        # recursive search from root
    # This is your capabilities audit command

Set a capability:
    setcap cap_net_bind_service=+ep /path/to/binary

Remove a specific capability:
    setcap cap_net_bind_service=-ep /path/to/binary

Remove ALL capabilities from a file:
    setcap -r /path/to/binary

View capabilities of running processes:
    cat /proc/PID/status | grep Cap
    # Shows CapInh, CapPrm, CapEff, CapBnd, CapAmb in hex
    # Decode with: capsh --decode=0000003fffffffff

SECTION 11 - EXTENDED FILE ATTRIBUTES (chattr/lsattr) IN DEPTH
================================================================

The original lists chattr flags. This explains what they do
and when to use them.

chattr flags that matter:

+i  immutable
    File cannot be modified, deleted, renamed, or linked.
    Even root cannot modify it without removing the flag first.
    Use: protect critical config files, /etc/passwd backups.

    chattr +i /etc/resolv.conf    # ISPs sometimes overwrite this
    chattr +i /etc/hosts          # prevent accidental overwrite
    lsattr /etc/resolv.conf       # shows: ----i---------e--- /etc/resolv.conf
    chattr -i /etc/resolv.conf    # remove immutable to edit

+a  append-only
    File can only be opened for appending, not overwriting.
    Perfect for log files -- even root cannot alter history.

    chattr +a /var/log/secure     # can write new lines, not modify old
    chattr +a /var/log/auth.log

+A  no atime updates
    Accessing the file does not update the access timestamp.
    Performance optimization -- prevents disk writes on every read.

+s  secure deletion
    When deleted, the blocks are zeroed. Prevents recovery.
    Note: on SSDs and journaling filesystems this is not
    guaranteed to work as intended. Use shred or cryptsetup
    for sensitive data instead.

+u  undeletable
    When deleted, kernel saves the contents for later recovery.
    Implementation depends on filesystem.

+e  extents
    File uses extents for block mapping (set automatically on ext4).

+C  no copy-on-write
    On btrfs and similar filesystems, disables COW for this file.
    Used for virtual machine disk images to reduce fragmentation.

+c  compressed
    File is automatically compressed by the kernel.
    Filesystem must support compression (btrfs, etc.).

Viewing attributes:
    lsattr file.txt            # show attributes for one file
    lsattr -R /etc             # recursive
    lsattr -d /etc             # show directory attributes (not contents)

Output format:
    ----i---------e--- /etc/resolv.conf
    The dashes are positions for each attribute flag.
    i = immutable, e = extents, all others empty here.

Important security note:
    chattr +i is not visible to standard tools (ls, stat).
    A file can look deletable but chattr +i prevents it.
    Always check lsattr when you cannot delete/modify a file
    even as root.

    # "Why can't I delete this file even as root?"
    lsattr /path/to/file       # check for +i flag

SECTION 12 - PROCESS PERMISSIONS AND EFFECTIVE UID
================================================================

Understanding why SUID works requires understanding how the
kernel checks permissions during process execution.

----------------------------------------------------------------
12a. Three UIDs every process has
----------------------------------------------------------------

Every running process has three user IDs:

Real UID (RUID):
    Who actually started the process. Your login UID.
    Stays constant throughout the process's life normally.

Effective UID (EUID):
    What the kernel uses for permission checks.
    Normally equals the real UID.
    Changed by SUID: when you run a SUID binary, EUID becomes
    the file owner's UID.

Saved Set-UID (SUID in process context):
    Saved copy of the original EUID before a privilege change.
    Allows a process to drop and regain privilege.

Example: running passwd (SUID root)
    alice runs /usr/bin/passwd
    RUID = 1001 (alice's UID)  <- who really started it
    EUID = 0    (root)         <- file owner's UID (SUID kicks in)
    passwd can now write to /etc/shadow because kernel sees EUID=0

Check a process's UIDs:
    cat /proc/PID/status | grep -E "^(R|E|S)uid"
    ps -o pid,ruid,euid,suid,comm

----------------------------------------------------------------
12b. How the kernel checks permissions
----------------------------------------------------------------

When a process tries to access a file, the kernel:

    1. Is EUID == 0 (root)?  YES -> allow everything
    2. Is EUID == file owner UID?  YES -> apply owner rwx bits
    3. Is any of process's GIDs == file's GID?  YES -> apply group rwx bits
    4. Apply others rwx bits

The kernel checks these in order and stops at the first match.

Important implication:
    If you OWN a file but have permission 007 (---rwxrwx),
    the owner bits (---) apply to you and you cannot read
    your own file, even though others can.

    chmod 007 myfile.txt
    cat myfile.txt     # Permission denied - you own it but 007 denies owner

    This trips up people who think "I own it so I can do anything."
    The kernel applies ownership category, not logic.

----------------------------------------------------------------
12c. Checking effective vs real UID in scripts
----------------------------------------------------------------

In a script, check if running as root:
    if [ "$(id -u)" -ne 0 ]; then
        echo "Must be run as root"
        exit 1
    fi

    # id -u returns EUID (effective), so this catches SUID escalation too

Check if SUID elevated your script (it won't, but shows the concept):
    echo "Real UID: $(id -ru)"        # real UID
    echo "Effective UID: $(id -u)"    # effective UID

SECTION 13 - chmod --reference AND chgrp
================================================================

Two commands that belong here and are absent.

chmod --reference:
    Copy permissions from one file to another.

    chmod --reference=template.sh newscript.sh
    # newscript.sh gets exactly the same permissions as template.sh

    stat template.sh    # shows 0755
    chmod --reference=template.sh newscript.sh
    stat newscript.sh   # now also 0755

    Useful when deploying scripts that must match existing permissions.

chgrp - change group owner only:
    The original uses chown :group for group changes.
    chgrp is the dedicated command for the same thing.

    chgrp developers file.txt       # change group to developers
    chgrp -R www-data /var/www/     # recursive
    chgrp --reference=ref.txt file.txt  # copy group from reference

    chgrp vs chown :group:
    chown :developers file.txt      # same result
    chgrp developers file.txt       # same result
    Both work. chgrp is more explicit about intent.

SECTION 14 - SECURITY AUDITING WITH PERMISSIONS
================================================================

Practical commands for finding permission problems on a system.

Find all SUID binaries (run periodically and compare):
    find / -perm -4000 -type f 2>/dev/null | sort > /tmp/suid_audit.txt
    diff /tmp/suid_baseline.txt /tmp/suid_audit.txt
    # Any new SUID binary since baseline is suspicious

Find world-writable files outside /tmp:
    find / -perm -0002 -type f ! -path "/proc/*" ! -path "/tmp/*" 2>/dev/null

Find files with no owner or group:
    find / -nouser -o -nogroup 2>/dev/null
    # These should not exist on a clean system
    # Appear when users are deleted without cleaning up files

Find files owned by root with write for others:
    find / -user root -perm -0002 -type f 2>/dev/null
    # Root-owned but world-writable = serious problem

Check all home directories have correct ownership:
    ls -la /home/              # all should be owned by their user
    stat /home/alice           # verify permissions are 700 or 750

Find recently modified SUID files (potential compromise):
    find / -perm -4000 -newer /var/log/syslog 2>/dev/null

List all files alice owns:
    find / -user alice 2>/dev/null

Verify /etc/passwd and /etc/shadow permissions:
    ls -la /etc/passwd     # should be -rw-r--r-- root:root 644
    ls -la /etc/shadow     # should be -r-------- root:shadow or 640
    ls -la /etc/sudoers    # should be -r--r----- root:root 440

SECTION 16 - COMMON PERMISSION PATTERNS AND WHAT THEY MEAN
================================================================

Reference for understanding permissions you encounter in the wild.

Files:
    -rw-------  600  owner read/write only. SSH private keys, root files.
    -rw-r--r--  644  standard file. Everyone reads, only owner writes.
    -rw-rw-r--  664  group can write. Collaborative files.
    -rw-rw-rw-  666  everyone can write. Temporary shared files. Use carefully.
    -rwx------  700  owner executable only. Personal scripts.
    -rwxr-xr-x  755  standard executable. Everyone can run it.
    -rwxrwxrwx  777  everyone can do everything. Almost never correct for files.
    -r--------  400  read-only even for owner. Backup files, old logs.
    -r--r--r--  444  everyone read-only. Published static files.

Directories:
    drwx------  700  private. Owner only. Typical user home (/home/user).
    drwxr-x---  750  owner full, group read/traverse. Shared with group only.
    drwxr-xr-x  755  standard directory. Everyone can list and traverse.
    drwxrwsr-x  2775 SGID + group write. Shared project directories.
    drwxrwxrwt  1777 World-writable with sticky. /tmp pattern.

Special files you'll see in ls -l:
    -rwsr-xr-x  4755  SUID. Runs as file owner (root usually).
    -rwxr-sr-x  2755  SGID on file. Runs as file's group.
    drwxrwsrwt  3777  SGID dir + sticky. Collaborative + protected.

Real system examples:
    /etc/passwd         -rw-r--r-- (644) root:root
    /etc/shadow         ---------- (000) or -r-------- (400) root:shadow
    /etc/sudoers        -r--r----- (440) root:root
    /home/alice         drwx------ (700) alice:alice
    /tmp                drwxrwxrwt (1777) root:root
    /usr/bin/passwd     -rwsr-xr-x (4755) root:root
    /usr/bin/sudo       -rwsr-xr-x (4755) root:root
    ~/.ssh/             drwx------ (700) user:user   MUST be 700
    ~/.ssh/authorized_keys -rw------- (600) user:user MUST be 600
    ~/.ssh/id_rsa       -rw------- (600) user:user   MUST be 600
    ~/.ssh/id_rsa.pub   -rw-r--r-- (644) user:user

SSH permission requirements are enforced by sshd:
    If ~/.ssh/ or key files have wrong permissions, SSH refuses
    to use them for security reasons.
    sshd will log: "bad permissions: ignore key file"

SECTION 17 - QUICK PERMISSION DEBUGGING WORKFLOW
================================================================

When you get "Permission denied" and need to fix it fast.

Step 1: Identify what is failing
    ls -la /path/to/file           # check the file itself

Step 2: Check all parent directories
    namei -l /path/to/file         # check entire path
    # Look for any directory that blocks your access

Step 3: Confirm your identity
    id                             # confirm your UID and groups
    # Newly added to a group? You need to log out and back in.

Step 4: Check for special blockers
    lsattr /path/to/file           # check for +i (immutable)
    getfacl /path/to/file          # check for ACL overrides

Step 5: Check process's effective UID (if a service is failing)
    systemctl status servicename   # check what user service runs as
    cat /proc/PID/status | grep Uid  # check running process's UID

Common fixes:
    # Wrong file ownership:
    chown correctuser:correctgroup /path/to/file

    # Wrong directory permissions:
    chmod 755 /path/to/directory

    # User not in required group:
    usermod -aG groupname username
    # (user must re-login after)

    # File is immutable:
    chattr -i /path/to/file        # remove immutable
    # then make your change
    chattr +i /path/to/file        # re-apply if needed

    # ACL is blocking:
    getfacl /path/to/file          # read what the ACL says
    setfacl -m u:yourusername:rw /path/to/file  # add yourself
    # or
    setfacl -b /path/to/file       # remove all ACLs, revert to standard
